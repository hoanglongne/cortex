🐛Các Bug Cần Sửa

🔴 MỨC ĐỘ CAO

1. Race Condition Khi Ghép Cặp

Vấn đề: Khi 2 user polling đồng thời, cả hai có thể tìm thấy nhau → tạo 2 match khác nhau → vào 2 phòng riêng biệt

Nguyên nhân: Logic matchmaking không atomic, không có database-level locking

Giải pháp: Tạo RPC function với transaction locking

2. Thiếu Rate Limiting

Vấn đề: User có thể spam nút "Find Match" liên tục → overload server

Giải pháp: Server-side cooldown 5s + client-side debounce

3. Band Matching Quá Rộng

Vấn đề: Hiện tại cho phép ±2.0 band ngay từ đầu → user band 5.0 có thể ghép với band 7.0

Giải pháp: Dynamic tolerance dựa trên thời gian chờ (bắt đầu ±0.5, max ±2.0)

Task 1.1: Fix Race Condition Trong Matchmaking
Tại sao quan trọng?
Đây là bug nghiêm trọng nhất hiện tại. Khi 2 user cùng tìm match, họ có thể bị đưa vào 2 phòng khác nhau → trải nghiệm tệ, user confusion.

Bước 1: Tạo Supabase RPC Function

File mới: supabase/migrations/YYYYMMDD_atomic_matchmaking.sql

CREATE OR REPLACE FUNCTION create_match_atomic(
  p_user1_id uuid,
  p_user2_id uuid
) RETURNS TABLE(match_id uuid, room_id text) AS $$
DECLARE
  v_match_id uuid;
  v_room_id text;
  v_existing_match uuid;
BEGIN
  -- Step 1: Lock cả 2 queue entries
  PERFORM * FROM match_queue 
  WHERE user_id IN (p_user1_id, p_user2_id) 
  AND status = 'waiting'
  FOR UPDATE NOWAIT;
  
  -- Step 2: Check nếu đã có match pending
  SELECT id INTO v_existing_match FROM matches
  WHERE (user1_id = p_user1_id OR user2_id = p_user1_id 
         OR user1_id = p_user2_id OR user2_id = p_user2_id)
  AND status = 'pending'
  LIMIT 1;
  
  IF v_existing_match IS NOT NULL THEN
    RAISE EXCEPTION 'User already has pending match';
  END IF;
  
  -- Step 3: Tạo match mới
  v_match_id := gen_random_uuid();
  v_room_id := 'room-' || v_match_id::text;
  
  INSERT INTO matches (id, user1_id, user2_id, room_id, status)
  VALUES (v_match_id, p_user1_id, p_user2_id, v_room_id, 'pending');
  
  -- Step 4: Update queue status
  UPDATE match_queue 
  SET status = 'matched' 
  WHERE user_id IN (p_user1_id, p_user2_id);
  
  -- Step 5: Return match info
  RETURN QUERY SELECT v_match_id, v_room_id;
END;
$$ LANGUAGE plpgsql;

Task 1.2: Thêm Rate Limiting

Server-side Rate Limiting trong findMatch()

export async function findMatch() {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return { success: false, error: 'Unauthorized' };
  
  // Check last request timestamp
  const { data: lastRequest } = await supabase
    .from('match_queue')
    .select('created_at')
    .eq('user_id', user.id)
    .order('created_at', { ascending: false })
    .limit(1)
    .single();
  
  if (lastRequest) {
    const timeSinceLastRequest = Date.now() - new Date(lastRequest.created_at).getTime();
    const cooldownMs = 5000; // 5 seconds
    
    if (timeSinceLastRequest < cooldownMs) {
      const remainingSeconds = Math.ceil((cooldownMs - timeSinceLastRequest) / 1000);
      return { 
        success: false, 
        error: `Vui lòng đợi ${remainingSeconds}s trước khi thử lại` 
      };
    }
  }
  
  // ... rest of matchmaking logic
}
Task 1.3: Dynamic Band Matching Tolerance
Công thức: Bắt đầu với ±0.5 band, tăng dần 0.5 mỗi 15s chờ, max ±2.0 
// Calculate wait time
const waitTimeSeconds = queueEntry.queue_joined_at 
  ? (Date.now() - new Date(queueEntry.queue_joined_at).getTime()) / 1000
  : 0;

// Dynamic tolerance
const baseTolerance = 0.5;
const incrementInterval = 15; // seconds
const maxTolerance = 2.0;

const tolerance = Math.min(
  baseTolerance + Math.floor(waitTimeSeconds / incrementInterval) * 0.5,
  maxTolerance
);
